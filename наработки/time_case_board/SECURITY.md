# Безопасный запуск под корпоративным VPN

Этот сервис рассчитан на **локальный** режим: браузер и backend только на **127.0.0.1**, без публикации в интернет и без обхода периметра.

## Что делает приложение

| Куда | Зачем | Когда |
|------|--------|--------|
| **127.0.0.1** | UI и API | Всегда |
| **TIME_BASE_URL** (TiMe) | OAuth, каналы, посты | Если пользуетесь вкладкой TiMe |
| **JIRA_BASE_URL** | REST API v2, Bearer-токен | Если заданы `JIRA_*` и открыта вкладка Jira |

Запросы к Jira и TiMe идут **с вашей рабочей станции** так же, как из браузера или PowerShell — **не через туннели и не «наружу»**, если ваш VPN/политика допускают прямой доступ к этим хостам.

## Чего не делать (чтобы не нарушать типовые регламенты)

1. **Не коммитьте** `oauth.env`, токены, Client Secret, `JIRA_TOKEN` — только локальный файл (уже в `.gitignore`).
2. **Не выставляйте** uvicorn на `0.0.0.0` в небезопасной сети без согласования ИБ.
3. **Не синхронизируйте** папку с `oauth.env` в облачные диски, если политика запрещает.
4. **Не прокидывайте** трафик через личные VPN/прокси «в обход» корпоративного доступа к Jira.
5. При ошибках SSL к Jira используйте **официальные** корневые сертификаты организации (`SSL_CERT_FILE` / доверенное хранилище ОС), а не отключение проверки, если ИБ это запрещает.

## Минимизация данных

- Вкладка **Jira** запрашивает список задач по JQL; данные **не сохраняются** в БД приложения, только отображаются в памяти браузера.
- Токены **не логируются** и не отдаются в `/api/jira/status`.

## Проверка под VPN

1. Подключите VPN по регламенту работодателя.
2. В PowerShell уже проверен `Invoke-WebRequest` к Jira — при необходимости повторите.
3. В UI вкладка **Jira** → **«Проверить Jira»** (эндпоинт `GET /api/jira/ping`, вызов `/rest/api/2/myself`).

Если ping не проходит — сначала сеть/сертификаты/токен, а не код приложения.
